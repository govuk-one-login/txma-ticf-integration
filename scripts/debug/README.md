# Local Lambda Debugging

Runs Lambda handlers directly in Node.js without Docker or SAM, enabling breakpoint debugging in VS Code and IntelliJ.

## Prerequisites

- [Node.js](https://nodejs.org/en/) version 24 (via [NVM](https://github.com/nvm-sh/nvm))
- `tsx` â€” installed via `npm install`
- [AWS CLI](https://docs.aws.amazon.com/cli/latest/userguide/install-cliv2.html) with AWS SSO configured
- [jq](https://jqlang.org/) â€” required by `fetch-env-vars.sh`
- Access to the dev AWS environment with the `AdministratorAccessPermission` SSO role

**Important**: Lambda execution roles can be assumed by developers with the `AdministratorAccessPermission` SSO role in the dev environment for debugging purposes. The debug script will attempt to assume the Lambda execution role to provide realistic permissions for testing operations like cross-account KMS decryption.

```bash
# Use the AdministratorAccessPermission SSO role for debugging
aws sso login --profile audit-dev
```

## Setup

Run once after cloning:

```bash
bash scripts/debug/setup/install-run-configs.sh
```

This generates `.vscode/launch.json` with a debug configuration for each Lambda. It also prints the manual steps required to set up IntelliJ run configurations.

### Environment variables

`env-vars.json` is the local substitute for Lambda environment variables. Fetch the current values from the deployed dev stack:

```bash
aws sso login --profile audit-dev
bash scripts/debug/setup/fetch-env-vars.sh
```

Re-run this whenever the deployed configuration changes (e.g. new queue URLs or table names). The script merges variables from all Lambda functions and applies local overrides (`XRAY_ENABLED=false`, etc.).

## Structure

- `invoke-local.ts` â€” entry point; loads `env-vars.json`, builds a mock Lambda context, and invokes the handler
- `events/<lambdaName>.json` â€” test event payload for each Lambda
- `env-vars.json` â€” environment variables injected at runtime (generated by `fetch-env-vars.sh`)
- `setup/install-run-configs.sh` â€” generates IDE debug configurations
- `setup/fetch-env-vars.sh` â€” fetches env vars from the deployed dev Lambda functions

## Adding a new Lambda

1. Create a test event: `scripts/debug/events/<lambdaName>.json`
2. Re-run setup to regenerate IDE configurations and fetch env vars (function names are discovered automatically from `template.yaml`):

```bash
bash scripts/debug/setup/fetch-env-vars.sh
bash scripts/debug/setup/install-run-configs.sh
```

The install script auto-discovers lambdas from `src/lambdas/` â€” no other changes needed.

## Notes

- Edit `events/<lambdaName>.json` to change the input event
- **Role Assumption**: The debug script attempts to assume the Lambda execution role for realistic permissions. This works for developers with the `AdministratorAccessPermission` SSO role in the dev environment. If role assumption fails, the script continues with your current credentials but may encounter permission issues.
- The `initiateDataRequest` handler validates the Zendesk webhook signature â€” expect a 400 response locally unless `ZENDESK_WEBHOOK_SECRET_KEY` is set correctly in `env-vars.json`
- The `initiateAthenaQuery` handler reads the raw `zendeskId` directly from `Records[0].body` â€” the event body is a plain string, not a JSON object
- The `dataReadyForQuery` and `sendQueryResultsNotification` handlers expect EventBridge events, not SQS events
- The `decryptAndCopy` handler expects an S3 Batch event and requires `ANALYSIS_BUCKET_NAME` to be set in `env-vars.json`
